Objetivo Funciones limpias

Qué problema resuelve

Cuando empiezas a crear varias funciones, necesitas saber qué variables existen dentro de cada una y cómo evitar dependencias confusas con el exterior.

Variables locales

Entender por qué una variable creada dentro no siempre existe fuera.

Claridad

Evitar funciones que dependen de cosas “escondidas”.

Mantenimiento

Escribir funciones más fáciles de entender y probar.

Diseño

Separar cálculo, impresión y decisiones en piezas más limpias.

Idea clave: una función suele ser mejor cuando depende de sus parámetros y no de variables externas escondidas.

Qué debes vigilar

  • una variable local suele existir solo dentro de la función
  • no mezclar demasiadas responsabilidades en una sola función
  • usar nombres que indiquen bien la intención
  • evitar duplicación cuando una función puede resolverla
Piensa así: una función clara no solo funciona; también se entiende rápido.

¿Qué es el ámbito?

El ámbito indica desde dónde puede usarse una variable. Una variable creada dentro de una función normalmente pertenece a esa función.

Variable local

def ejemplo():
    mensaje = "Hola"
    print(mensaje)

ejemplo()

mensaje existe dentro de la función.

Error típico: usar fuera una variable local

def ejemplo():
    mensaje = "Hola"

print(mensaje)

Esto falla porque mensaje no existe fuera del ámbito de la función.

Variables externas y parámetros

En lugar de depender de variables externas, normalmente conviene pasar datos como parámetros. 

def saludar(nombre):
    print("Hola", nombre)

Funciones pequeñas y claras

Una función no debería hacer demasiadas cosas a la vez. Cuanto más concreta sea su responsabilidad, más fácil será entenderla y reutilizarla.

Ejemplo comparado: función confusa vs función limpia

def hacer_cosa(x):
    print("Resultado:", x * 1.21)

Funciona, pero su nombre no ayuda y mezcla cálculo con impresión. 

def calcular_iva(precio):
    return precio * 0.21

Esta versión expresa mejor la intención y devuelve un valor útil.

Nombres correctos

def calcular_descuento(precio):
    return precio * 0.9

Este nombre deja bastante clara la intención. En cambio, def hacer_cosa(x): no ayuda nada.

Evitar duplicación

Si descubres que repites el mismo bloque de código dos o más veces, probablemente conviene convertirlo en función.

Separar cálculo de presentación

Una técnica profesional muy útil es separar la función que calcula de la que imprime. 

def calcular_iva(precio):
    return precio * 0.21

precio = 100
iva = calcular_iva(precio)
print("IVA:", iva)

Responsabilidades mezcladas

Calcular

La función obtiene un resultado.

Mostrar

Otra parte del programa lo enseña por pantalla.

Decidir

Otra función o bloque puede tomar decisiones con ese resultado.

Ventaja

El código se vuelve más claro, reutilizable y fácil de probar.

Cómo pensar una función limpia antes de escribirla

Paso 1

Decide qué responsabilidad exacta va a tener la función.

Paso 2

Evita que dependa de variables externas si puedes pasar esos datos por parámetros.

Paso 3

Elige un nombre que explique bien lo que hace.

Paso 4

Separa calcular, decidir y mostrar cuando eso mejore la claridad.

Una función limpia no solo resuelve el problema. También deja claro cómo lo resuelve y para qué sirve.

Errores frecuentes

Error 1

Crear funciones demasiado largas.

Error 2

Usar nombres poco descriptivos.

Error 3

Depender demasiado de variables externas.

Error 4

Mezclar cálculo, impresión y decisiones en una sola función.

Buenas prácticas

Responsabilidad clara

Cada función debería resolver una tarea principal, no muchas a la vez.

Nombres útiles

El nombre debería indicar qué hace la función sin necesidad de leerla entera.

Parámetros antes que dependencias

Mejor pasar datos por parámetros que depender de variables externas ocultas.

Separación de tareas

Si una función calcula, intenta que otra parte del programa se encargue de mostrar.

Resumen mental del subtema

Qué debes tener claro antes de seguir.

Ideas clave

  • una variable local existe solo dentro de su función
  • las funciones claras dependen de parámetros, no de variables “escondidas” fuera
  • un buen nombre comunica mejor la intención que un comentario confuso
