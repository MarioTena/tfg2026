Qué vas a entender hoy

Un error no es solo “algo que sale mal”: es una señal que te dice en qué parte del programa hay un problema y de qué tipo es.

Objetivo Entender fallos

Qué problema resuelve

Hasta ahora has trabajado muchas veces en condiciones normales. Pero en la práctica real también hay conversiones inválidas, archivos que no existen, divisiones imposibles o reglas mal planteadas.

Entradas inválidas

Un usuario puede escribir algo inesperado.

Archivos ausentes

El programa puede intentar abrir algo que no existe.

Índices incorrectos

Puedes intentar acceder a una posición que no existe.

Lógica errónea

El programa puede funcionar, pero devolver un resultado mal planteado.

Idea clave: no todos los fallos son iguales. Entender su tipo cambia la forma de corregirlos.

Qué debes vigilar

  • no todos los fallos son de sintaxis
  • algunos aparecen al ejecutar, no al escribir
  • otros no rompen el programa, pero producen resultados incorrectos
  • leer bien el mensaje de error ayuda mucho más que asustarse
Piensa así: un mensaje de error no es un ataque, es información útil para localizar el problema.

Dónde aparecen los errores en problemas reales

Entradas inválidas

Un usuario puede escribir algo distinto de lo esperado.

Archivos ausentes

El programa puede intentar abrir algo que no existe.

Índices o claves incorrectas

Puedes intentar acceder a datos que no están disponibles.

Lógica mal planteada

El programa puede ejecutarse, pero producir un resultado incorrecto.

Los errores aparecen en problemas reales porque los datos, las condiciones y el entorno no siempre son perfectos.

Por qué este tema importa de verdad

Hasta ahora has aprendido a escribir programas que funcionan en condiciones normales. Pero en la práctica, los programas reales también deben saber reaccionar cuando algo sale mal.

Un principiante intenta que nunca haya errores. Un programador más sólido asume que algunos fallos ocurrirán y aprende a interpretarlos y gestionarlos.

Qué es un error

Un error es una situación en la que el programa no puede continuar normalmente o no produce el comportamiento correcto que se esperaba.

Puede deberse a código incorrecto, datos no válidos, archivos ausentes o lógica mal planteada.

Error general y excepción

En este tema hablarás mucho de excepciones. Una excepción es un tipo de error que aparece durante la ejecución del programa y que Python puede señalar con un nombre concreto, como ValueError o FileNotFoundError.

Regla mental útil: “error” es la idea general; “excepción” es el problema concreto que Python detecta mientras ejecuta el programa.

Tipos generales de problemas

Error de sintaxis

El programa ni siquiera puede interpretarse.

Error en ejecución

Aparece mientras el programa ya está corriendo.

Error lógico

El programa corre, pero hace algo incorrecto.

Lectura correcta

No todos se corrigen del mismo modo.

Error de sintaxis

if 5 > 3
    print("Hola")

Aquí falta :. Python no puede ni empezar a ejecutar el programa.

Error en tiempo de ejecución

numero = int("hola")

El programa empieza, pero falla al intentar convertir un texto no numérico a entero.

Error lógico

precio = 100
descuento = 10
final = precio - descuento / 100
print(final)

El programa funciona, pero el cálculo no representa el descuento que seguramente se pretendía.

Error del programador frente a datos externos

Error del programador

Una variable mal escrita o una sintaxis incorrecta.

Dato externo problemático

El usuario escribe algo inválido o el archivo no existe.

Error lógico

La fórmula o la decisión están mal pensadas.

Importancia

No todos se resuelven igual: a veces corriges código y otras controlas entradas.

Algunas excepciones muy comunes en Python

ValueError

El valor tiene un formato inválido.

TypeError

Estás mezclando tipos incompatibles.

NameError

La variable no existe.

IndexError

El índice está fuera del rango válido.

KeyError

La clave no existe en el diccionario.

FileNotFoundError

El archivo que intentas abrir no existe.

ZeroDivisionError

Intentas dividir entre cero.

Idea útil

El nombre de la excepción ya te da una pista fuerte sobre el problema.

Gestionar errores no es lo mismo que ignorarlos

Gestionar errores significa detectar situaciones problemáticas y responder de forma controlada, en lugar de dejar que el programa se rompa sin más.

La gestión de errores no elimina todos los problemas, pero evita respuestas bruscas, confusas o poco útiles.

Cómo pensar un error antes de corregirlo

Paso 1

Mira si el programa ni siquiera arranca.

Paso 2

Si arranca, revisa en qué línea falla.

Paso 3

Lee el tipo de error antes de tocar el código.

Paso 4

Piensa si el fallo es técnico, de entrada o lógico.

Piensa antes de ejecutar

Antes de lanzar el código, intenta clasificar el problema en tu cabeza.

Código

edad = int("veinte")

Preguntas

  • ¿funcionará?
  • ¿qué tipo de problema provoca?
  • ¿por qué esa conversión no puede hacerse?
Pista: aquí el programa sí arranca, pero falla al ejecutar una conversión concreta.
