Misión 8.4

Cuándo usar cada estructura

Aprende a elegir entre lista, tupla o set según lo que el problema realmente necesita representar.

Qué vas a entender hoy

Elegir bien entre lista, tupla o set no es un detalle menor: cambia cómo representas el problema y cómo se entiende tu código.

Objetivo Elegir bien

Qué problema resuelve

No basta con que el programa funcione. También importa que la estructura elegida represente bien lo que estás modelando: datos ordenados, fijos, únicos o modificables.

Lista

Para secuencias ordenadas que pueden cambiar y admitir repetidos.

Tupla

Para grupos ordenados que deberían mantenerse estables.

Set

Para valores únicos donde importa la pertenencia, no la posición.

Diseño

Elegir bien mejora claridad, intención y mantenimiento del código.

Idea clave: la estructura correcta no se elige por costumbre, sino por la naturaleza del problema.

Qué debes vigilar

  • si el orden importa
  • si puede haber duplicados
  • si los datos van a cambiar
  • si lo importante es pertenencia o acceso por posición
Piensa así: antes de escribir código, pregúntate qué representa mejor tus datos en el mundo real.

Dónde aparece elegir estructuras en problemas reales

Pedidos

Usar lista si importa el orden y puede haber productos repetidos.

Coordenadas

Usar tupla cuando el grupo es fijo y no debería cambiar.

Etiquetas

Usar set si lo importante es evitar duplicados y comprobar pertenencia.

Diseño real

Elegir bien la estructura hace el código más claro y representa mejor el problema.

En problemas reales no eliges una estructura porque “sirve”, sino porque representa mejor lo que necesitas modelar.

La decisión correcta importa

Elegir bien la estructura de datos es una habilidad importante. No se trata solo de que el programa funcione, sino de que represente bien el problema.

Usa lista cuando...

  • importa el orden
  • admites duplicados
  • vas a modificar, añadir o eliminar con frecuencia
  • la posición forma parte del problema
tareas = ["estudiar", "practicar", "repasar"]

Usa tupla cuando...

  • importa el orden
  • el grupo de datos es fijo o estable
  • no quieres expresar cambios frecuentes
  • cada posición tiene un significado estable
coordenada = (40.4, -3.7)

Usa set cuando...

  • no quieres duplicados
  • el orden no importa para el problema
  • quieres comprobar pertenencia
  • quieres comparar grupos de elementos
categorias = {"python", "web", "datos"}

Comparación rápida

Lista

Orden + cambios + repetidos posibles.

Tupla

Orden + estabilidad.

Set

Unicidad + pertenencia.

Regla útil

No empieces por la sintaxis: empieza por el comportamiento que necesitas.

Casos comparativos claros

Notas de una clase

Suele encajar mejor una lista si importa el orden o puede haber repetidos.

Latitud y longitud

Suele encajar mejor una tupla porque es un par fijo y ordenado.

Usuarios únicos conectados

Suele encajar mejor un set porque no quieres duplicados y te importa la pertenencia.

Idea clave

No todas las colecciones resuelven igual de bien el mismo caso.

Casos ambiguos donde hay que pensar mejor

Asientos numerados

¿Te importa el orden y la posición? Entonces un set no suele ser buena idea.

Etiquetas de un artículo

Si no quieres repetidos, un set suele representar mejor el problema que una lista.

Horario fijo de apertura y cierre

Una tupla puede expresar bien un par estable de datos.

Lista de compra editable

Una lista suele encajar mejor si vas a cambiar elementos y el orden importa.

Cuando una estructura “funciona” pero representa peor

A veces puedes usar una lista donde una tupla sería más expresiva, o una lista donde un set modelaría mejor el problema. Que el código funcione no significa que sea la mejor elección.

Elegir bien la estructura ya forma parte del diseño del programa.

Cómo decidir correctamente

Paso 1

Pregunta si el orden forma parte del problema.

Paso 2

Pregunta si puede haber duplicados.

Paso 3

Pregunta si esos datos deben cambiar después.

Paso 4

Pregunta si lo importante es la pertenencia o la posición.

Piensa antes de ejecutar

Aquí no se trata de ejecutar código, sino de razonar qué estructura encaja mejor en cada caso.

Situaciones

  • notas de una clase
  • posición GPS
  • emails únicos recibidos
  • lista de tareas editable

Preguntas

  • ¿en cuáles importa el orden?
  • ¿en cuáles quieres evitar duplicados?
  • ¿cuál representa mejor datos fijos?
  • ¿en cuál la pertenencia importa más que la posición?
